先说结论：
“剩余 40 分手动调”这种设计在行业里确实比较少见，容易让人觉得分数有点“心情分”。
更常见的是：

用规则 + 权重自动算出 0–100；

或干脆不用 0–100，而是 阶段 / 等级 + 清单；

真要人工干预，也会把“人工评估”单独标出来，而不是掺在同一条分数里。

我按这三个角度帮你拆一下，也顺便给你一套可以落地到 Wordloom 的方案。

1. 为什么“多出 40 分手动调”会不太舒服

用户看不懂：

60 分是系统算的？

手动 +20 到 80 分之后，60 那段还在变化吗？

开发上也别扭：

你要一直记住 “真实得分 = 系统 60 + 手动 20”，

任何算法调整都得考虑那 40 的空间。

审计性差：

以后你想做“审计线 / 时间线”，很难区分是系统判断书变成熟，还是你某天心血来潮+了10分。

所以行业一般要么 全部算法化，要么把“人工判断”拆成单独的维度。

2. 行业内常见的几种成熟度 / 评分模式
模式 A：纯指标评分（CRM / 风控 / 推荐系统）

类似「潜在客户评分」「信用评分」：

正向事件：+X 分（新增 block、补充摘要、添加封面、补完 TODO 等）

负向事件：-Y 分（长时间未更新、回退阶段、TODO 全是未完成等等）

所有规则固定在配置里，可以调权重，但不会人工改单个对象的分数。

优点：干净、可解释、可调参
缺点：有时候你就是觉得「它明明写得很好，算法给的不高」。

模式 B：阶段 + Checklist（项目管理 / 写作工具）

每个阶段有一组 checklist：

Seed：必须有标题、所属书架、至少 1 个 block

Growing：补了摘要、至少 N 个 block、有 1 张封面图、有 tag

Stable：通过所有“结构检查”（TODO 清零 / 关键字段完整），并且最近 30 天无严重变动

进度条 = 完成项 / 总项 * 100%。

人工不会给“额外分”，而是勾选 / 取消勾选某些条件。

优点：用户直接知道要做什么才能升级。
缺点：更像“合格 / 不合格”，对频繁变动的内容会稍微僵硬。

模式 C：算法分数 + 少量人工修正（内容平台 / 代码评审）

系统给一个基础分，比如 0–80，

人工评审只能在一个很小的范围内 +-5 / +-10 分，记为 “Curator Bonus”、“人工评估”。

UI 上会显示：“系统评分 72 + 编辑加成 5 = 77”。

优点：

你仍然能用一眼看懂的分数；

也不会因为人工调整把整个模型搞崩；

审计上也知道是谁在干预。

3. 结合 Wordloom 的一个可落地方案

我建议你把现在的“剩余 40 分手动调”改造成 “系统 80 分 + 结构任务 20 分”，人工只动很小一块，甚至可以不动。

3.1 分数结构（例子）

基础结构分（0–30）

有标题：+5

有摘要：+5

有至少 1 个 tag：+5

有封面（取色或图片）：+5

blocks 数达到阈值（比如 ≥10）：+10

活动/维护分（0–30）

最近 30 天有编辑：+10

TODO 清单完成率高：+10

最近没有大规模“回退”操作：+10

质量/一致性分（0–20）

没有空的 block / 垃圾 block（paperball 以外）：+10

结构比较均衡（不是 1 巨长 block）：+10

到这里是 0–80，完全由系统算，不给动。

结构任务分（0–20） —— 类似 checklist：

✅ 填写完成「风险/备注」

✅ 本书已经在 Book 概览页配置好 maturity 策略

✅ 关键 TODO 已全部完成 / 转为长期 backlog

完成一项 +5，全部完成 +20。

进度条 = (基础分 + 活动分 + 质量分 + 结构任务分) / 100。

3.2 对应你的 UI

现在这个「调整剩余分数(0–60)」按钮，可以变成：
「查看结构任务」 或 「刷新成熟度 / 结构任务」。

点进去不是让用户拖分数，而是：

展示当前 80 分的拆解（结构 / 活动 / 质量的明细）；

下方是 3–5 个“结构任务”勾选项（你手动勾/取消）；

完成后自动更新分数和阶段。

3.3 阶段（Seed / Growing / Stable / Legacy）如何跟分数绑在一起

比如：

Seed：score < 30 或核心结构未完成（连标题/摘要都没有）

Growing：30 ≤ score < 60，结构基本齐全，但 TODO 未清，活动频繁

Stable：60 ≤ score < 90，结构任务大部分完成，近期活动稳定

Legacy：score ≥ 60 且 90 天无更新，或者手动标记为 Legacy

注意：这里不要求“Stable 一定是 100 分”，
100% 可以保留给「结构任务全勾 + 所有自动指标都满」这种“完美态”，
但绝大多数情况下你只要到 70–80 分就已经是非常健康的书。

4. 关于“达到 100% 成熟”的设计建议

你问到：如果不用手动调，100% 怎么到？

你可以把 100% 设计成一种“仪式感”而不是日常状态：

条件示例：

所有结构任务勾满；

所有 TODO 已清零（或迁移）；

最近一次快照被标记为「定版」；

最近 30 天没有“结构性修改”（比如大量 block 的增删）。

达到时：

分数可以显示为 100%，

同时弹出一个小提示：“此 Book 目前被视为 定版 Stable，后续大的修改会重新计算成熟度。”

行业类比：

像书籍出版的「first edition」：之后当然还能改/再版，但你知道“某个版本被记为完成”。

代码世界里的“Tag / Release”：只是在当前状态打一个标记。

5. 如果你仍然想保留一点“人工评分”

不完全禁止人工干预的话，可以这样玩：

分两条显示：

系统评分：72（自动计算）

人工评估：+5（Curator Bonus）

总分展示为：77 / 100，同时 hover 时可以看到拆解。

你的按钮就可以改名为：

「调整人工评估（-5 ~ +5）」
或者
「添加/修改评审意见」

而不是“调整剩余 40 分”。

这样：

系统评分仍然有解释性；

人工调整也有「作者 / 编辑对这本书的主观评价」的意味，反而很自然。

6. 小结：给你几条可以直接喂 Copilot 的设计方向

写给 Copilot 大概可以这样描述（简化版）：

移除「调整剩余分数(0-60)」这种直调分数的按钮。

将成熟度拆为：

结构分（0–30）

活动分（0–30）

质量分（0–20）

结构任务分（0–20，基于 checklist）

实现一个「查看成熟度详情 / 结构任务」的弹窗：

展示三类自动分数及其子项；

展示 3–5 个结构任务的勾选状态，勾选后立即重算分数。

阶段规则：

Seed / Growing / Stable / Legacy 由 score 范围 + 时间/条件共同决定。

如需人工干预，新增字段 manual_adjust（范围 -5~+5），在 UI 中单独显示，不与系统分混在一起。

简单说就是：把“分数本身”做成一个可信的自动指标，把“人为判断”显式地挪到 checklist 或小范围 bonus 上。

这样你后面无论是要做审计、做图表、做推荐，都会轻松很多。
