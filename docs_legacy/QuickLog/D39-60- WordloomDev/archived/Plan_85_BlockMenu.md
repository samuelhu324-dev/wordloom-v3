你这个菜单现在的问题，其实是一个典型的「指针要跑半个屏幕」+「菜单太长」的问题。

先说结论：
行业里不会让“插入块菜单”变成一条从天顶拉到地狱的长条。
常见做法有三种，你可以组合用：

0. 立刻能缓解的「止疼药」

在不改交互的大前提下，先让现在这个菜单好用一点：

给菜单本身加 max-height + overflow-y: auto

.block-insert-menu {
  max-height: 280px;     /* 自己调一个舒服的值 */
  overflow-y: auto;
}


这样菜单出现的位置不变，但内部可以滚动，不会“顶到上面去了，底下还要追着鼠标跑”。

把高频放上面 / 低频折叠到“更多”里

你现在有：

段落

Todo 列表

Callout

引用

分割线

（以后还有图片、图片组、代码……）

很现实的一个判断：
80% 的时间你只会用 3–5 种类型（段落、Todo、Callout、分割线、图片）。

所以可以直接改结构：

高频（总是展开）
  段落
  Todo 列表
  Callout
  分割线

其他（一个“更多块类型…”行，点开再展开剩下的列表）


这样你现在这个长单子立即会短一半，鼠标移动距离就没那么夸张了。

1. 行业常见方案 A：Notion 式「/ 命令 + 就地菜单」

逻辑：菜单出现在光标附近，而不是页面某个固定位置。

用户在当前段落里输入 /；

在光标附近弹出一个小菜单（高度有限，内部可滚动）；

支持搜索，例如输入 /to 时过滤出 Todo List；

上下键 + Enter 选中，基本不需要用鼠标。

优点：

指针/视线都不需要大幅移动；

菜单可以很长，但通过搜索过滤，心智负担低；

完全贴合你现在的「无 block 感」编辑器。

给你的落地建议：

现在先保留下面这个 + 插入块 按钮；

之后可以加一个简单的 / command：

监听当前 textarea 的输入；

捕获 / 开头的一小段字符串；

在 caret 附近渲染一个 mini-menu，沿用你现在的分组+样式即可；

选中后，把 /xxx 那一段删掉，插入对应 block。

这套做完，你几乎就和 Notion 一样专业了。

2. 方案 B：局部的「行内 + 按钮」而不是底部大菜单

很多编辑器会把「插入」做成**“跟着当前行走”**的：

每一行左边有一个小的 + 或 ⋯ 图标（只有 hover 时出现）；

点这个 + 在这一行附近弹出一个小菜单（同样 max-height + scroll）；

这个菜单只管在这一行插什么。

你现在底部已经有 + 添加一段文字 / + 插入块，
可以改成：

「全局添加」还是保留在底部（用于在最后追加内容）；

但真正频繁用的「在中间插一块」使用行内 + 来完成。

这样：

鼠标只是在你正在看的那一块附近小范围移动；

底部那个长菜单主要负责“最后再加东西”，不会成为主要入口。

3. 方案 C：少量固定工具 + “更多”弹窗 / 侧栏

还有一派做法是：

底部/顶部有一条常驻工具条，只放图标 + 少数常用类型：

段落、Todo、Callout、分割线、图片（最多 5 个）；

旁边有一个「⋯ / 更多块类型」按钮：

点了之后，弹出一个 大一点的「块库」对话框，用网格展示所有 block 类型；

这个东西使用频率一般不高，所以即便对话框大一点、需要移动鼠标，也可以接受。

优点：

主编辑体验非常干净：永远只有一小排图标；

新增奇怪类型（比如将来的图表、代码块）直接塞进「更多」就好了，不会污染主工具条。

你可以这样组合现在的设计：

底部保留：
+ 添加一段文字（永远插段落）
右边加一小排 icon（Todo / Callout / Divider / 图片）；

再加一个 ⋯ 更多…，点击才弹出你现在这个「完整块列表」。

4. 给你的实际行动顺序（不折腾心理能量版）

如果按投入产出比排个顺序，我建议：

现在就做的：

给现有菜单加 max-height + overflow-y；

把高频类型固定在上半部分，低频折叠到「更多块类型…」。

接下来一两天可以做的：

把底部的 + 插入块 简化，只展示高频类型（图标或短文字）；

把“完整块菜单”挂在「更多」按钮上，不再每次都拉一大长条。

等 Book / Block 总体稳定后：

实验 / 命令菜单（光标附近的小菜单，配合键盘操作）；

或者做行内的 + 插入。

这样你既不会把自己 UI 搞得太复杂，又能和行业里那几款“严肃 Notion / 文档器”站在同一代设计里。

简单一句话概括就是：

常用的放在手边，少用的藏到「更多」+ 可滚动菜单 / 对话框里，
菜单尽量贴着光标而不是贴着屏幕边缘。
