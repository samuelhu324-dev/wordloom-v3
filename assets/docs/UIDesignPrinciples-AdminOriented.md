# 二、Admin 主题评分标准 v1（内核狠辣版）

> 面向 **后台 / 运维 / 审计 / 高级用户界面**，要求：诚实、暴露问题、不拯救面子。
> 气质：严肃、可追责，像博物馆工作台 + 工业控制室。

## 0. 统一打分区间

所有维度分数 `Aᵢ ∈ {0,1,2,3,4,5}`：

- **0 分**：信息严重缺失 / 完全不适合运营使用，有安全或事故风险
- **1 分**：极差，只能靠人工补救或猜测维持运转
- **2 分**：不足，勉强能用，但大量关键问题被藏起来
- **3 分**：合格，日常可用，但遇到异常就非常吃力
- **4 分**：专业，能支撑重度使用和异常处理
- **5 分**：卓越，适合当“审判厅”：任何问题都有迹可循、责任可追

---

## 1. 维度与权重

### A₁：可观测性与真相暴露度（w₁ = 1.7）

- 当前系统状态（成功 / 失败 / 延迟 / 阻塞）是否一眼看出？
- 关键指标是否在首页 / 核心仪表板上清晰呈现，而不是藏在三级页面？
- 是否避免“假平静”：界面看起来一切正常，但实际问题被隐藏？

> **5 分标准：**
> “一眼知道今天有没有问题；如果有，知道大概在哪块。”

---

### A₂：风险可见性与告警设计（w₂ = 1.5）

- 错误、异常、边缘状态是否有明确标记，而不是静悄悄？
- 告警是否有优先级区分（致命 / 重要 / 提醒），避免告警雪崩？
- 危险操作（删除数据、重算、迁移）是否有扎实的保护机制和清晰说明？

> **5 分标准：**
> “所有危险都有明确黄/红线，没有‘一不小心就炸库’的暗坑。”

---

### A₃：审计追踪与责任链（w₃ = 1.6）

- 关键操作是否有日志：谁在何时做了什么，对什么对象？
- 是否提供按时间 / 对象 / 操作者的过滤和回溯能力？
- 出事时，是否能在 5–10 分钟内查到“来源、影响范围、改动细节”？

> **5 分标准：**
> “任何改动都像抽屉里的档案：翻得到、看得懂、说得清是谁干的。”

---

### A₄：信息架构与复杂度承载力（w₄ = 1.4）

- 在对象多、层级深时，结构是否仍然清晰、有路可循？
- 是否避免把不同概念混在一个列表里，让人分不清维度？
- 命名、分区、导航是否坚持统一规则，而不是随手一页一个世界？

> **5 分标准：**
> “再复杂的数据，也能按逻辑拆成几块稳定的‘展柜’。”

---

### A₅：高频操作效率与批处理能力（w₅ = 1.2）

- 针对运营常用任务（批量修改、筛选、导出、对账），是否有专门优化？
- 是否支持键盘操作 / 快捷方式、批量选择、智能默认值？
- 是否尽量减少“重复跳回列表再进入详情再返回”的往返？

> **5 分标准：**
> “重度用户可以在这里‘像打命令行一样快’，而不是被 UI 拖慢。”

---

### A₆：异常场景与灾难恢复（w₆ = 1.1）

- 是否覆盖：无数据 / 部分失败 / 外部服务不可用 / 权限不足 等边缘状态？
- 错误提示是否说明“是什么 + 为什么 + 建议下一步”？
- 有无基本的“撤销 / 反向操作 / 只读模式”，避免一次误操作不可挽回？

> **5 分标准：**
> “出了问题不是靠人扛，而是系统先给出一条理性的解决路径。”

---

### A₇：氛围与心理预期管理（w₇ = 0.8）

- 视觉风格是否传达“严肃、冷静、工具感”，而非花哨或娱乐？
- 重要区块（统计卡片、日志流、过滤条）是否像“展柜 + 铭牌”，
  让人自然严肃起来？
- 是否避免用俏皮文案掩饰严重问题（例如：生产事故还配可爱 emoji）？

> **5 分标准：**
> “一进后台就知道：这里不是玩笑场，是工作台和档案室。”
